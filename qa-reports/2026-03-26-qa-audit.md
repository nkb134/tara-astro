**Root Cause**: `handlePlace()` in `onboardingHandler.js` passes the raw `messageText` (with no length/sentence check) directly to `generateChartFromPlace()`. **Fix needed** (`src/services/onboardingHandler.js`, `handlePlace` function, ~line 418):
```js

// If input looks like a full sentence/question, re-ask for place

  return { response: t(lang, 'ask_place'), messageType: 'simple' };
}
```
Also fix the disambiguation template in all language files to NOT include `{city}` (the user-provided text) at the start — use a generic phrase instead:

### BUG-2: "Samay-2pm" Parsed as Place Name (CRITICAL)
**Severity**: Critical — causes onboarding to stall permanently

**What happened** (User 18, Nihar, 2026-03-24 20:56):
- User was in `awaiting_time` step
- User sent: *"Samay-2pm"* (Hindi: "Time is 2pm")
- Tara responded: *"Ok, Samay, Bong County — iss location ke hisaab se kundli nikal rahi hoon"*
- "Samay" (meaning "time" in Hindi) was treated as a city name

**Root Cause**: In `handleTime()`, when time parsing returns something, the system advances to `awaiting_place`. But in this case, it seems the message was NOT processed by `handleTime` as expected — likely because "Samay-2pm" didn't match the time parser's patterns (expects "2pm", "2:00 PM", etc. but not "Samay-2pm"). The message then fell through to `generateChartFromPlace` which geocoded "Samay" as a location.

**Fix needed** (`src/services/onboardingHandler.js`):
- The time parser needs to handle `Samay-<time>`, `time:<time>`, `waqt:<time>` patterns
- Also: if the `awaiting_time` handler can't parse a time from the message, it should re-ask for time, NOT fall through to generating a chart from a garbled location
